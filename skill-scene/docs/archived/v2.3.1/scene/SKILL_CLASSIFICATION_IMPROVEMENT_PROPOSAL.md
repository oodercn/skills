## 一、问题背景

### 1.1 当前分类设计

当前 `skill-index.yaml` 中定义了 17 个分类，面向普通用户的分类仅 3 个：

| 分类ID | 名称 | 可见性 | 面向用户 |
|--------|------|--------|:--------:|
| llm | AI助手 | PUBLIC | ✅ |
| know | 知识服务 | PUBLIC | ✅ |
| util | 工具服务 | PUBLIC | ✅ |
| org | 组织服务 | DEVELOPER | ❌ |
| vfs | 存储服务 | DEVELOPER | ❌ |
| ... | ... | ... | ... |

### 1.2 发现的问题

**问题1: 业务场景无处安放**

| 业务场景 | 当前归属 | 问题 |
|----------|----------|------|
| HR人力资源 | 无对应技能 | 普通用户无法找到 |
| 工作日志 | 无对应技能 | 普通用户无法找到 |
| 审批流程 | 无对应技能 | 普通用户无法找到 |
| 客户管理(CRM) | 无对应技能 | 普通用户无法找到 |
| 财务管理 | 无对应技能 | 普通用户无法找到 |
| 项目协作 | 无对应技能 | 普通用户无法找到 |

**问题2: 分类过于技术化**

当前面向普通用户的分类 (`llm`, `know`, `util`) 都是技术导向的命名，用户难以理解：
- 用户不知道 "LLM" 是什么
- 用户不知道 "知识服务" 包含什么
- 用户不知道 "工具服务" 有哪些

**问题3: Scenes 与 Categories 不匹配**

Scenes 中定义了 `category: BUSINESS`，但 Categories 中没有对应的 `business` 分类：

```yaml
# scenes 中存在
- sceneId: hr
  name: Human Resources
  category: BUSINESS  # ← 但 categories 中没有 business

- sceneId: crm
  name: Customer Relationship Management
  category: BUSINESS  # ← 但 categories 中没有 business
```

## 二、修改建议

### 2.1 新增"业务场景"分类

建议在 `categories` 中新增 `biz` 分类，面向普通用户：

```yaml
- id: biz
  name: 业务场景
  nameEn: Business
  description: 人力资源、客户管理、财务审批、项目管理等业务场景
  icon: briefcase
  order: 4
  address: "0x48-0x4F"
  visibility: PUBLIC
  ownership: SIC
  userFacing: true
  displayGroup: 业务场景
```

### 2.2 定义子分类体系

建议为 `biz` 分类定义子分类：

| 子分类ID | 名称 | 描述 | 示例技能 |
|----------|------|------|----------|
| hr | 人力资源 | 员工管理、考勤、薪酬、招聘 | skill-hr-employee, skill-hr-attendance |
| crm | 客户管理 | 客户、联系人、商机、合同 | skill-crm-customer, skill-crm-contact |
| finance | 财务管理 | 账务、报销、预算、结算 | skill-finance-expense, skill-finance-budget |
| approval | 审批流程 | 流程定义、审批、流转 | skill-approval-leave, skill-approval-expense |
| project | 项目协作 | 项目、任务、文档、进度 | skill-project-manage, skill-project-task |
| worklog | 工作日志 | 日报、周报、工时统计 | skill-worklog-daily, skill-worklog-weekly |

### 2.3 修改后的普通用户分类

```
普通用户可见分类 (4个):
├── 🤖 AI助手 - 智能对话、代码助手、写作助手
├── 📚 知识服务 - 知识问答、文档检索、知识库管理
├── 💼 业务场景 - HR、财务、审批、项目、工作日志
│   ├── 人力资源 - 员工管理、考勤打卡、薪资查询、招聘助手
│   ├── 客户管理 - 客户管理、联系人、商机管理
│   ├── 财务管理 - 报销审批、预算管理、账务查询
│   ├── 审批流程 - 请假审批、报销审批、采购审批
│   ├── 项目协作 - 项目管理、任务分配、进度跟踪
│   └── 工作日志 - 日报填写、周报汇总、工时统计
└── 🛠️ 工具服务 - 技能市场、报表、分享
```

## 五、备选方案

### 5.1 方案A：保持现状，扩展 util 分类

将业务场景技能放入 `util` 分类：

**优点**: 无需新增分类，改动最小  
**缺点**: `util` 分类过于臃肿，用户难以区分工具和业务场景

### 5.2 方案B：重命名现有分类

将 `llm` 重命名为 `ai-assistant`，将 `know` 重命名为 `knowledge`：

**优点**: 分类名称更友好  
**缺点**: 需要修改现有代码引用，兼容性风险

### 5.3 推荐方案

**推荐方案A + 新增 biz 分类**：
1. 新增 `biz` 分类用于业务场景
2. 保持现有分类不变，确保向后兼容
