# SDK 协作说明

## 协作背景

Nexus Enterprise 项目需要构建完整的能力管理界面，但在实际开发过程中发现 SDK 层存在关键能力缺失。本文档说明协作目标、现状和期望。

---

## 协作目标

建立 SDK 与上层应用之间的标准事件通信机制，使外部系统能够实时感知 Skill 的生命周期状态变化。

---

## 现状分析

### 已验证的 SDK 状态

| 检查项 | 状态 | 验证方式 |
|--------|------|----------|
| Skill 静态配置读取 | ✅ 可用 | 读取 `skill.yaml` 文件 |
| Skill 列表查询 | ✅ 可用 | `/api/test/capability/skills/list` |
| Skill 详情查询 | ✅ 可用 | `/api/test/capability/skills/detail` |
| 运行时状态查询 | ⚠️ Mock 数据 | 返回硬编码数据 |
| 生命周期事件订阅 | ❌ 缺失 | 黑盒探测未找到 |
| 健康检查机制 | ❌ 缺失 | 无主动健康上报 |
| 能力调用链路追踪 | ❌ 缺失 | 无事件产生 |

### 核心问题

SDK 目前仅提供**静态配置读取**能力，缺乏**运行时事件通知**机制。外部系统无法通过观察者模式监听以下关键事件：

- Skill 创建/加载/启动/停止/卸载
- 能力调用开始/完成/失败
- 健康状态变化
- 错误和异常

---

## 协作内容

### 1. SDK 需提供的能力

#### 1.1 事件类型定义

SDK 需定义标准的生命周期事件类型枚举，覆盖 Skill 从创建到销毁的完整生命周期。

#### 1.2 观察者接口

SDK 需提供观察者接口，允许外部系统订阅特定 Skill 或特定类型的事件。

#### 1.3 事件发布服务

SDK 需提供事件发布服务，在 Skill 状态变化时主动推送事件给订阅者。

#### 1.4 Spring Boot 集成

SDK 需提供 Spring Boot Starter，实现自动配置和依赖注入。

### 2. 上层应用将提供的能力

#### 2.1 外部观察者实现

在 SDK 提供原生事件机制之前，上层应用已实现基于轮询的外部观察者作为过渡方案：

- **文件**: `net.ooder.nexus.observer.SkillLifecycleObserver`
- **功能**: 定时轮询 API，检测 Skill 状态变化
- **局限**: 非实时，有延迟，资源消耗较高

#### 2.2 测试验证 API

上层应用提供测试 API 用于验证 SDK 事件机制：

- `POST /api/test/observer/report` - 获取观察报告
- `POST /api/test/observer/trigger/capability` - 触发能力调用并观察
- `POST /api/test/observer/check/events` - 检查事件类型支持

#### 2.3 能力管理界面

上层应用将基于 SDK 提供的事件机制，构建完整的能力管理界面：

- Skill 生命周期可视化
- 实时状态监控
- 事件时间线展示
- 告警和通知

## 协作流程

### Phase 1: 需求确认（1-3 天）

1. SDK 团队评审协作请求文档
2. 双方确认事件类型和接口定义
3. 确定技术方案和实现细节

### Phase 2: SDK 开发（1-2 周）

1. SDK 团队实现生命周期事件机制
2. 提供 Beta 版本供上层应用测试
3. 迭代修复问题

### Phase 3: 集成验证（3-5 天）

1. 上层应用集成 SDK Beta 版本
2. 使用测试 API 验证事件机制
3. 性能测试和压力测试

### Phase 4: 正式发布（1 周）

1. SDK 发布正式版本
2. 上层应用切换至原生事件机制
3. 废弃外部观察者过渡方案

## 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| SDK 开发延期 | 能力管理界面无法按时交付 | 外部观察者方案作为 fallback |
| 接口定义变更 | 双方需返工 | 需求确认阶段充分评审，冻结接口 |
| 性能不达标 | 事件延迟或丢失 | 增加性能测试环节，设置 SLA |
| 兼容性问题 | 与现有系统冲突 | 提供迁移指南，支持平滑升级 |
