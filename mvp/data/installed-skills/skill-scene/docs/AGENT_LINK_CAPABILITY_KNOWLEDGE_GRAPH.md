# Agent-Link-Capability 知识图谱与重构方案

## 一、核心概念定义

### 1.1 Agent（智能体）

```
Agent = 执行者 + 状态 + 能力

Agent {
  agentId: string          // 唯一标识
  name: string             // 名称
  type: AgentType          // 类型: LLM | WORKER | DEVICE | PLATFORM
  status: AgentStatus      // 状态: CREATED | INITIALIZING | RUNNING | PAUSED | STOPPED | FAILED
  capabilities: string[]   // 拥有的能力列表
  metadata: object         // 元数据
}
```

**Agent 类型说明**：
| 类型 | 说明 | 典型场景 |
|------|------|----------|
| LLM | 大语言模型 Agent | 对话生成、内容创作 |
| WORKER | 工作 Agent | 任务执行、数据处理 |
| DEVICE | 设备 Agent | IoT 设备控制 |
| PLATFORM | 平台 Agent | 系统服务、基础设施 |

**SuperAgent（超级智能体）**：
```
SuperAgent = Agent[] + Coordination + EmergentCapabilities

SuperAgent {
  superAgentId: string
  subAgents: SubAgentRef[]      // 子 Agent 列表
  coordination: CoordinationConfig  // 协调配置
  emergentCapabilities: Capability[]  // 涌现能力
  selfDefined: boolean          // 是否自驱
}
```

### 1.2 Link（链路）

```
Link = Source + Target + Type + Status

Link {
  linkId: string            // 链路ID
  sourceId: string          // 源 Agent ID
  targetId: string          // 目标 Agent ID
  type: LinkType            // 类型: DIRECT | RELAY | TUNNEL | MULTICAST | P2P
  status: LinkStatus        // 状态: ACTIVE | INACTIVE | DEGRADED | FAILED | PENDING
  latency: number           // 延迟(ms)
  bandwidth: number         // 带宽
}
```

**Link 类型选择规则**：
| 场景 | Link 类型 | 说明 |
|------|-----------|------|
| 同域内 Agent 通信 | DIRECT | 低延迟，无中间节点 |
| 跨域通信（有网关） | RELAY | 通过中间节点转发 |
| 私有域访问公共域 | TUNNEL | 加密隧道，需权限验证 |
| 一对多通知 | MULTICAST | 组播，高效广播 |
| 北向 P2P 通信 | P2P | WebRTC 等去中心化 |

### 1.3 Capability（能力）

```
Capability = 定义 + 归属 + 可见性

Capability {
  capabilityId: string         // 全局唯一标识
  name: string                 // 名称
  type: CapabilityType         // 类型: SERVICE | DRIVER | ORCHESTRATION | SCENE
  ownership: CapabilityOwnership  // 归属: PLATFORM | INDEPENDENT | SCENE_INTERNAL
  visibility: string           // 可见性: public | internal | private
  supportedSceneTypes: string[]  // 支持的场景类型
  parentSkill: string          // 父技能ID
  parentScene: string          // 父场景ID
}
```

**能力三层模型**：
```
┌─────────────────────────────────────────────────────────────┐
│ 第一层：静态能力定义（模板层）                                │
│   Capability (能力定义)                                      │
│   ├── capabilityId: 全局唯一标识                             │
│   ├── name, type, version: 基本属性                          │
│   └── accessLevel: 访问级别                                  │
├─────────────────────────────────────────────────────────────┤
│ 第二层：能力实例化（运行层）                                  │
│   能力实例 = Agent + Link + Address                          │
│   ├── Agent: 能力执行者                                      │
│   ├── Link: 通信链路                                         │
│   └── Address: CAP地址（权限控制）                            │
├─────────────────────────────────────────────────────────────┤
│ 第三层：能力调用（执行层）                                    │
│   CapabilityBinding (能力绑定)                               │
│   ├── capabilityId + capId: 双ID映射                         │
│   ├── agentId + linkId: 执行者和链路                         │
│   └── status: 绑定状态                                       │
└─────────────────────────────────────────────────────────────┘
```

## 八、核心洞察：场景组实例化引擎

### 8.1 关键问题

**问题**：场景组建立时究竟应该实例化哪些能力？

**示例**：
```
场景定义声明：需要 PDF-转文本 支持
场景组实例化：邀请特定地址的能力加入（如 192.168.1.100:8080/pdf-service）
```

**核心矛盾**：
- **能力定义**：静态声明，描述"需要什么能力"
- **能力实例**：运行时实体，描述"谁来提供这个能力"
- **能力绑定**：将能力实例绑定到场景组

## 九、关键决策点

### 9.1 需要讨论的问题

1. **Agent 与场景的关系**
   - Agent 是否需要绑定到特定场景？
   - 一个 Agent 能否服务多个场景？

2. **Link 的生命周期**
   - Link 是随 Agent 注册自动建立？
   - 还是按需手动建立？

3. **能力绑定的审批流程**
   - 哪些能力需要审批？
   - 审批人如何确定？

4. **能力隔离的粒度**
   - 场景组级别隔离？
   - 还是场景级别隔离？

### 9.2 建议方案

1. **Agent 与场景**：Agent 独立于场景，通过 CapabilityBinding 关联
2. **Link 生命周期**：Agent 注册时自动建立基础 Link，按需建立专用 Link
3. **审批流程**：INDEPENDENT 能力需要审批，PLATFORM 能力自动绑定
4. **能力隔离**：场景组级别隔离，SCENE_INTERNAL 能力仅在场景组内可见
