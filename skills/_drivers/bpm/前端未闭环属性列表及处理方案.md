## 四、处理方案

### 4.1 方案一：完全替换为新模型 (推荐)

**实施步骤：**

1. **替换模型类**
   - 使用 `ActivityDefNew.js` 替换 `ActivityDef.js`
   - 使用 `ProcessDefNew.js` 替换 `ProcessDef.js`

2. **更新面板配置**
   - 使用 `ActivityPanelSchemaNew.js` 替换 `ActivityPanelSchema.js`

3. **引入校验器**
   - 引入 `BpmJsonSchema.js` 进行数据校验

4. **更新 Store.js**
   - 在保存前调用 `BpmJsonSchema.Validator.validateProcessDef()`
   - 使用 `BpmJsonSchema.Transformer.toBackend()` 转换数据

5. **数据迁移**
   - 旧数据通过 `fromJSON` 方法自动转换
   - 属性组自动创建，缺失字段使用默认值

**优点：**
- 彻底解决所有未闭环问题
- 代码结构清晰，易于维护
- 有完整的测试覆盖

**风险：**
- 需要全面测试现有功能
- 可能需要调整部分业务逻辑

### 4.2 方案二：渐进式改造 (低风险)

**实施步骤：**

1. **第一阶段：修复关键属性 (P0)**
   - 在现有 `ActivityDef.js` 中添加 `participantId`
   - 在现有 `ProcessDef.js` 中添加 `startNode` 和 `endNodes`
   - 添加 `rightGroups` 支持

2. **第二阶段：添加属性组支持 (P1)**
   - 在 `ActivityDef.js` 中添加 RIGHT/FORM/SERVICE/WORKFLOW 属性组
   - 保持向后兼容，扁平属性映射到属性组

3. **第三阶段：完善其他属性 (P2)**
   - 添加流程级别的时限配置
   - 添加版本信息字段

**优点：**
- 风险低，可以分步实施
- 每次改动范围小，易于测试

**缺点：**
- 代码会暂时存在两套逻辑
- 改造周期较长

### 4.3 方案三：适配层方案 (临时方案)

**实施步骤：**

1. **创建数据适配器**
   ```javascript
   class DataAdapter {
       // 前端数据 -> 后端格式
       static toBackend(frontendData) {
           // 转换属性组结构
           // 转换数据类型
           // 添加缺失字段
       }
       
       // 后端数据 -> 前端格式
       static fromBackend(backendData) {
           // 反向转换
       }
   }
   ```

2. **在 API 层使用适配器**
   - 发送前调用 `toBackend`
   - 接收后调用 `fromBackend`

**优点：**
- 无需修改现有模型
- 快速实现数据对齐

**缺点：**
- 增加代码复杂度
- 只是临时方案，技术债务增加
