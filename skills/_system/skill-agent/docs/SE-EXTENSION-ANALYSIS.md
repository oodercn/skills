# 方案对比分析：扩展 SE SDK vs SPI 封装

> **文档版本**: v1.0 | **日期**: 2026-04-05

---

## 方案定义

### 方案 C: 扩展 SE SDK

在 scene-engine SDK 中补充缺失的类型和接口，使其能够被业务层直接使用。

**具体操作**:
1. 在 SE SDK 中新增 `MessageParticipant` 类
2. 调整 `MessageEnvelope.setFrom/setTo` 接受 String 或提供工厂方法
3. 简化 `NorthboundMessageQueue` 的方法签名
4. 发布 SE SDK 3.0.2 或 3.1.0

---

## 详细对比

### 1. 架构层面

| 维度 | 扩展 SE SDK | SPI 封装 |
|------|------------|---------|
| **分层清晰度** | ⚠️ 模糊：SE 同时承担基础设施+业务接口 | ✅ 清晰：SE 是基础设施，SPI 是业务接口 |
| **依赖方向** | ⚠️ 业务层直接依赖 SE SDK | ✅ 业务层依赖 SPI，实现层依赖 SE |
| **职责单一** | ⚠️ SE 职责扩大 | ✅ 各层职责明确 |

### 2. 开发效率

| 维度 | 扩展 SE SDK | SPI 封装 |
|------|------------|---------|
| **短期工时** | ✅ 较少：改 SE + 改调用方 | ⚠️ 较多：新增 SPI + 实现层 |
| **长期维护** | ⚠️ SE 变更影响所有调用方 | ✅ SE 变更仅影响实现层 |
| **调试复杂度** | ⚠️ 需要理解 SE 内部实现 | ✅ 业务层只看 SPI |

### 3. 版本管理

| 维度 | 扩展 SE SDK | SPI 封装 |
|------|------------|---------|
| **SE 发版频率** | ⚠️ 需要频繁发版适配业务需求 | ✅ SE 独立演进，不受业务影响 |
| **兼容性** | ⚠️ SE 接口变更导致编译失败 | ✅ SPI 稳定，实现层适配 |
| **回滚风险** | ⚠️ 高：SE 回滚影响所有模块 | ✅ 低：仅实现层回滚 |

### 4. 团队协作

| 维度 | 扩展 SE SDK | SPI 封装 |
|------|------------|---------|
| **职责边界** | ⚠️ SE 团队需理解业务需求 | ✅ SE 团队专注基础设施 |
| **并行开发** | ⚠️ 业务团队等待 SE 发版 | ✅ 业务团队先定义 SPI，并行开发 |
| **代码审查** | ⚠️ SE 变更需多方评审 | ✅ 各层独立审查 |

## 扩展 SE SDK 的具体影响

### 优点

| # | 优点 | 说明 |
|---|------|------|
| 1 | **减少抽象层** | 业务层直接调用 SE，无额外封装 |
| 2 | **性能无损** | 无中间层转换，调用链短 |
| 3 | **代码量少** | 不需要定义 SPI 接口和实现类 |
| 4 | **快速见效** | 改完 SE 即可使用 |

### 缺点

| # | 缺点 | 说明 | 风险等级 |
|---|------|------|---------|
| 1 | **SE 职责膨胀** | SE 本是"场景引擎"，现在还要做"消息服务" | 🔴 高 |
| 2 | **耦合严重** | 所有业务模块直接依赖 SE 具体类型 | 🔴 高 |
| 3 | **版本地狱** | SE 任何接口变更都导致业务编译失败 | 🟡 中 |
| 4 | **测试困难** | 业务单元测试需要 mock SE 类型 | 🟡 中 |
| 5 | **团队瓶颈** | SE 团队成为所有业务需求的第一道关卡 | 🟡 中 |
| 6 | **无法替换** | 未来想换消息中间件，所有业务代码都要改 | 🔴 高 |

## 场景模拟

### 场景 1: SE SDK 需要重构 MessageEnvelope

**扩展 SE SDK 方案**:
```
SE 团队修改 MessageEnvelope
    → 所有依赖 SE 的模块编译失败
    → 需要协调所有模块同步修改
    → 发布顺序必须严格: SE 先，业务后
    → 回滚困难
```

**SPI 封装方案**:
```
SE 团队修改 MessageEnvelope
    → 仅 MessagingServiceImpl 需要适配
    → 业务层代码无需修改
    → 各模块独立发布
    → 回滚简单
```

### 场景 2: 业务需要新增消息类型

**扩展 SE SDK 方案**:
```
业务团队提出需求
    → SE 团队评估排期
    → SE 发版
    → 业务团队升级 SE 依赖
    → 业务开发
```

**SPI 封装方案**:
```
业务团队定义 SPI 接口
    → 业务团队开发
    → 实现层适配 SE (可并行)
    → 独立发布
```

### 场景 3: 需要替换消息中间件

**扩展 SE SDK 方案**:
```
所有 import net.ooder.scene.message.* 的代码都要改
    → 改动量巨大
    → 风险极高
    → 可能需要数周
```

**SPI 封装方案**:
```
仅实现层需要改
    → 业务层代码不动
    → 改动量小
    → 风险可控
```

## 混合方案建议

如果选择"扩展 SE SDK"，建议采用**混合方案**降低风险：

```
┌─────────────────────────────────────────────────────────────────┐
│                    混合方案架构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  SE SDK 3.1.0 (扩展版)                                           │
│  ├─ 新增 MessageParticipant 类                                  │
│  ├─ MessageEnvelope 增加 from(String)/to(String) 便捷方法       │
│  └─ NorthboundMessageQueue 增加 send(sceneGroupId, Map) 方法    │
│                                                                 │
│  ooder-spi-core (薄封装)                                         │
│  ├─ MessagingService 接口 (可选，推荐保留)                       │
│  └─ 默认实现: 直接委托给 SE SDK                                  │
│                                                                 │
│  skill-agent                                                    │
│  └─ 可选择:                                                      │
│      A. 直接用 SE SDK (快速但有耦合)                             │
│      B. 用 SPI 接口 (推荐，解耦)                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 混合方案实施步骤

| 步骤 | 任务 | 工时 |
|------|------|------|
| 1 | SE SDK 新增 `MessageParticipant` 类 | 0.5h |
| 2 | SE SDK `MessageEnvelope` 增加便捷方法 | 0.5h |
| 3 | SE SDK 发布 3.1.0 | 自动 |
| 4 | ooder-spi-core 新增 `MessagingService` | 0.5h |
| 5 | ooder-spi-core 提供默认实现 (委托 SE) | 0.5h |
| 6 | skill-agent 改用 SPI 接口 | 1h |

**总工时**: 约 3h

## 最终建议

### 推荐方案: SPI 封装 (得分 4.1)

**理由**:
1. 长期维护成本最低
2. 架构最清晰
3. 灵活性最高
4. 符合依赖倒置原则

### 次选方案: 混合方案 (得分 3.8)

**适用场景**:
1. 时间紧迫，需要快速见效
2. 团队对 SE SDK 熟悉
3. 愿意承担一定技术债务

### 不推荐: 纯扩展 SE SDK (得分 1.9)

**风险**:
1. 耦合严重，未来重构困难
2. SE 团队成为瓶颈
3. 版本管理复杂
