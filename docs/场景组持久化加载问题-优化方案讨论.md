## 一、问题背景

### 1.1 当前问题现象

场景组管理页面无法显示已持久化的场景组数据，日志显示：

```
Loading persisted scene groups...
Found 0 persisted scene groups
Scene group loading complete. Total groups in memory: 0
```

### 1.2 当前修复方案

修改 `SeCorePackages.PACKAGES` 列表，添加新的核心包：

```java
public static final List<String> PACKAGES = Arrays.asList(
    // ... 已有配置
    "net.ooder.scene.group.persistence.",  // 新增
    "net.ooder.scene.group."               // 新增
);
```

### 1.3 当前方案的问题

**核心问题**：这个职责不应该由 hotplug 模块每次都更新配置

**具体问题**：

| 问题 | 说明 |
|------|------|
| **违反开闭原则** | 每次新增核心包都需要修改 hotplug 模块代码 |
| **职责不清** | hotplug 模块不应该知道具体有哪些业务核心包 |
| **维护成本高** | 两处同步（SE 的 PluginClassLoader + hotplug 的 SeCorePackages） |
| **扩展性差** | 新模块无法自主声明核心包 |

## 三、优化方案设计

### 3.1 方案对比

| 方案 | 核心思路 | 优点 | 缺点 | 推荐度 |
|------|---------|------|------|--------|
| **方案一：模块自声明** | 每个模块在 META-INF 中声明核心包 | 符合开闭原则、模块自包含 | 需要修改 PluginClassLoader | ⭐⭐⭐⭐⭐ |
| **方案二：配置文件驱动** | application.yml 集中配置 | 集中管理、无需改代码 | 配置分散、不够内聚 | ⭐⭐⭐ |
| **方案三：接口约定** | 约定特定包前缀为核心包 | 简单、无配置 | 限制命名自由度 | ⭐⭐⭐⭐ |

### 3.2 推荐方案：模块自声明机制

**核心思想**：让模块自己声明哪些包需要从父类加载器加载，而非集中配置。

#### 3.2.1 声明文件格式

每个模块在 `META-INF/se-core-packages` 文件中声明：

```properties
# META-INF/se-core-packages
# 格式：每行一个包前缀
net.ooder.scene.group.persistence.
net.ooder.scene.group.
```

#### 3.2.2 实现方式

**Step 1：修改 PluginClassLoader**

```java
public class PluginClassLoader extends URLClassLoader {
    
    private static volatile Set<String> dynamicCorePackages = new HashSet<>();
    
    static {
        loadDynamicCorePackages();
    }
    
    private static void loadDynamicCorePackages() {
        Set<String> packages = new HashSet<>();
        
        // 从 classpath 中所有 JAR 加载 META-INF/se-core-packages
        try {
            Enumeration<URL> resources = 
                PluginClassLoader.class.getClassLoader()
                    .getResources("META-INF/se-core-packages");
            
            while (resources.hasMoreElements()) {
                URL url = resources.nextElement();
                try (InputStream is = url.openStream();
                     BufferedReader reader = new BufferedReader(
                         new InputStreamReader(is, StandardCharsets.UTF_8))) {
                    
                    String line;
                    while ((line = reader.readLine()) != null) {
                        line = line.trim();
                        if (!line.isEmpty() && !line.startsWith("#")) {
                            packages.add(line);
                        }
                    }
                }
            }
        } catch (IOException e) {
            logger.warn("Failed to load dynamic core packages", e);
        }
        
        dynamicCorePackages = Collections.unmodifiableSet(packages);
        logger.info("Loaded {} dynamic core packages: {}", packages.size(), packages);
    }
    
    private boolean shouldLoadFromParent(String name) {
        // 检查静态配置
        if (SeCorePackages.isSeCoreClass(name)) {
            return true;
        }
        
        // 检查动态配置（模块自声明）
        for (String pkg : dynamicCorePackages) {
            if (name.startsWith(pkg)) {
                return true;
            }
        }
        
        // ... 其他检查
    }
}
```

**Step 2：模块添加声明文件**

scene-engine 模块添加：

```
# 文件：scene-engine/src/main/resources/META-INF/se-core-packages
net.ooder.scene.group.persistence.
net.ooder.scene.group.
net.ooder.scene.spi.
net.ooder.scene.skill.
net.ooder.scene.llm.context.
```

**Step 3：简化 SeCorePackages**

```java
public final class SeCorePackages {
    
    // 仅保留框架级别的核心包（不变）
    public static final List<String> PACKAGES = Arrays.asList(
        // 框架级接口（极少变化）
        "net.ooder.scene.spi."
    );
    
    // 业务核心包由各模块通过 META-INF/se-core-packages 声明
}
```

### 3.3 方案优势

| 优势 | 说明 |
|------|------|
| **符合开闭原则** | 新增核心包无需修改 hotplug 模块 |
| **模块自包含** | 核心包声明与模块代码放在一起 |
| **类似 SPI 机制** | 开发者熟悉 META-INF 声明方式 |
| **易于维护** | 每个模块管理自己的声明 |
| **向后兼容** | 静态配置仍然有效 |

## 五、备选方案

### 5.1 方案二：配置文件驱动

在 `application.yml` 中配置：

```yaml
ooder:
  hotplug:
    se-core-packages:
      - net.ooder.scene.group.persistence.
      - net.ooder.scene.group.
```

**优点**：集中管理，无需修改代码
**缺点**：配置分散，不够内聚

### 5.2 方案三：接口约定

约定所有需要共享的接口必须放在特定的包前缀下：

```
net.ooder.scene.spi.*      # 场景引擎 SPI
net.ooder.skill.common.spi.* # 技能通用 SPI
```

**优点**：简单，无配置
**缺点**：限制命名自由度

## 六、总结

### 6.1 推荐方案

**方案一：模块自声明机制**

### 6.2 核心改进

| 改进点 | 说明 |
|--------|------|
| **职责分离** | hotplug 模块不再负责维护业务核心包列表 |
| **模块自治** | 每个模块自己声明需要共享的包 |
| **开闭原则** | 新增核心包无需修改框架代码 |

### 6.3 影响范围

| 影响项 | 说明 |
|--------|------|
| hotplug 模块 | 需要修改 PluginClassLoader |
| scene-engine | 需要添加 META-INF/se-core-packages |
| skill-common | 需要添加 META-INF/se-core-packages |
| 其他模块 | 可选添加声明文件 |
