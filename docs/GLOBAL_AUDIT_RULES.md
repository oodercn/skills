## 一、数据完整性验证规则

### 规则1.1：完整读取文件

**问题描述**: 只读取文件的部分内容，导致统计数据不准确

**错误案例**:
```
错误操作: 使用 Read 工具时只读取了部分内容（使用 offset/limit）
正确操作: 完整读取整个文件，或使用 Grep 统计所有匹配项
```

**审计规则**:
| 规则ID | 规则描述 | 验证方法 |
|--------|----------|----------|
| R1.1.1 | 禁止截断读取配置文件 | 必须完整读取 .yaml, .xml, .json 等配置文件 |
| R1.1.2 | 列表数据必须计数验证 | 使用 `wc -l` 或 Grep 统计数量 |
| R1.1.3 | 多Controller端点统计 | 搜索所有 `- path:` 或 `@Mapping` 注解 |

**验证命令模板**:
```bash
# 统计YAML中的端点数量
grep -c "^\s*- path:" skill.yaml

# 统计Java中的Mapping注解
grep -c "@.*Mapping" Controller.java

# 完整读取文件
cat skill.yaml | wc -l  # 确认总行数
```

### 规则1.2：多维度数据验证

**问题描述**: 只从一个来源获取数据，可能遗漏信息

**审计规则**:
| 规则ID | 规则描述 | 验证方法 |
|--------|----------|----------|
| R1.2.1 | 端点数量双重验证 | 同时检查 skill.yaml 和 Controller.java |
| R1.2.2 | 路由注册状态验证 | 同时检查日志和实际API响应 |
| R1.2.3 | 依赖版本验证 | 同时检查 pom.xml 和 maven 依赖树 |

## 二、版本信息验证规则

### 规则2.1：区分版本类型

**问题描述**: 混淆项目版本和依赖版本

**错误案例**:
```xml
<!-- 错误理解 -->
<artifactId>apex-os</artifactId>
<version>1.0.0</version>  <!-- 这是项目版本 -->

<properties>
    <ooder.sdk.version>3.0.1</ooder.sdk.version>  <!-- 这是依赖版本 -->
</properties>

<!-- 错误操作: 将 ooder.sdk.version 误认为项目版本 -->
```

**审计规则**:
| 规则ID | 规则描述 | 验证方法 |
|--------|----------|----------|
| R2.1.1 | 项目版本识别 | 读取 `<artifactId>` 后紧跟的 `<version>` 标签 |
| R2.1.2 | 依赖版本识别 | 读取 `<properties>` 中的版本属性 |
| R2.1.3 | 父项目版本识别 | 读取 `<parent>` 中的版本 |

**验证命令模板**:
```bash
# 获取项目版本
grep -A1 "<artifactId>apex-os</artifactId>" pom.xml | grep "<version>"

# 获取所有版本属性
grep -E "version>" pom.xml | head -20

# 使用maven验证实际版本
mvn help:effective-pom | grep -A1 "<artifactId>apex-os</artifactId>"
```

### 规则2.2：版本一致性检查

**审计规则**:
| 规则ID | 规则描述 | 验证方法 |
|--------|----------|----------|
| R2.2.1 | 父子项目版本一致性 | 检查子模块是否正确继承父项目版本 |
| R2.2.2 | 依赖版本范围 | 检查是否存在版本冲突 |
| R2.2.3 | SNAPSHOT版本警告 | 标记所有 SNAPSHOT 版本 |

## 三、JAR包验证规则

### 规则3.1：完整验证所有JAR包

**问题描述**: 只验证部分JAR包，假设其他JAR包状态相同

**错误案例**:
```
错误操作: 看到日志中几个JAR包的错误，假设所有相关JAR包都有问题
正确操作: 使用脚本验证所有JAR包
```

**审计规则**:
| 规则ID | 规则描述 | 验证方法 |
|--------|----------|----------|
| R3.1.1 | 批量验证所有JAR包 | 使用循环脚本检查所有JAR包 |
| R3.1.2 | 内容完整性检查 | 检查必需文件是否存在 |
| R3.1.3 | 时间戳对比 | 对比JAR包修改时间和错误日志时间 |

**验证命令模板**:
```bash
# 批量检查JAR包中的skill.yaml
Get-ChildItem plugins/*.jar | ForEach-Object {
    $name = $_.Name
    $hasYaml = (jar -tf $_.FullName 2>&1 | Select-String "skill.yaml").Count
    Write-Output "$name : $hasYaml"
}

# 检查JAR包修改时间
Get-ChildItem plugins/*.jar | Select-Object Name, LastWriteTime

# 检查JAR包内容
jar -tf plugin.jar | grep -E "skill.yaml|META-INF|\.class"
```

### 规则3.2：时间相关性验证

**审计规则**:
| 规则ID | 规则描述 | 验证方法 |
|--------|----------|----------|
| R3.2.1 | 错误时间记录 | 记录日志中错误发生的时间 |
| R3.2.2 | JAR包构建时间 | 检查JAR包的最后修改时间 |
| R3.2.3 | 时间先后判断 | 如果JAR包修改时间晚于错误时间，错误可能已修复 |

## 六、常见错误模式库

### 模式1：部分读取遗漏

**特征**:
- 统计数量偏少
- 只看到文件的前半部分
- 遗漏了后续的同类数据

**检测方法**:
- 检查是否使用了 offset/limit 参数
- 对比文件总行数和读取行数
- 使用 Grep 统计总数

### 模式2：属性混淆

**特征**:
- 版本号错误
- 名称相似但含义不同
- 嵌套属性理解错误

**检测方法**:
- 检查 XML/JSON 层级结构
- 区分元素内容和属性值
- 验证属性的作用域

### 模式3：时间错位

**特征**:
- 历史错误被当作当前问题
- 已修复的问题仍被报告
- 文件修改时间早于验证时间

**检测方法**:
- 对比日志时间和文件修改时间
- 检查是否有重新构建
- 验证当前运行状态

## 八、规则应用示例

### 示例1：端点数量审计

**错误做法**:
```
1. 读取 skill.yaml 前100行
2. 统计端点数量为8
3. 报告: 该模块有8个端点
```

**正确做法**:
```
1. 完整读取 skill.yaml
2. 使用 grep -c "^\s*- path:" 统计数量
3. 对比 Controller.java 中的 @Mapping 注解数量
4. 报告: 该模块有16个端点 (来源: skill.yaml 第67-171行, 验证时间: 2026-04-03 14:30)
```

### 示例2：版本信息审计

**错误做法**:
```
1. 看到 <ooder.sdk.version>3.0.1</ooder.sdk.version>
2. 报告: 项目版本为3.0.1
```

**正确做法**:
```
1. 读取 <artifactId>apex-os</artifactId> 后的 <version>1.0.0</version>
2. 区分项目版本(1.0.0)和SDK依赖版本(3.0.1)
3. 报告: 项目版本为1.0.0, SDK依赖版本为3.0.1
```

### 示例3：JAR包状态审计

**错误做法**:
```
1. 看到日志中5个JAR包的错误
2. 报告: 5个JAR包缺少skill.yaml
```

**正确做法**:
```
1. 使用脚本验证所有JAR包
2. 对比JAR包修改时间和日志错误时间
3. 发现只有1个JAR包当前缺少skill.yaml
4. 报告: 1个JAR包缺少skill.yaml (skill-scenes-1.0.0.jar), 其他4个已在12:12重新构建
```
