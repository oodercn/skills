## 一、用户逻辑分析

### 1.1 用户是谁？

根据需求规格，场景模块的用户分为三类：

| 用户类型 | 角色 | 发起动作 | 典型场景 |
|----------|------|----------|----------|
| **User** | 普通用户 | 创建场景、使用能力、配置参数 | 员工提交日志、用户配置智能家居 |
| **Agent** | 智能代理 | 自动执行任务、响应事件 | 定时提醒、数据处理、设备控制 |
| **SuperAgent** | 超级代理 | 协调多Agent、涌现能力 | LLM协调、跨场景协作 |

### 1.2 用户发起的动作

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        用户动作发起链路                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   User 发起:                                                            │
│   ├── 选择场景模板 → 部署场景                                           │
│   ├── 创建场景 → 配置能力 → 激活场景                                     │
│   ├── 绑定能力 → 配置参数                                               │
│   ├── 调用能力 → 获取结果                                               │
│   └── 管理参与者 → 分配角色                                             │
│                                                                         │
│   Agent 发起:                                                           │
│   ├── 定时触发 → 执行工作流                                             │
│   ├── 事件响应 → 调用能力                                               │
│   └── 数据处理 → 自动上报                                               │
│                                                                         │
│   SuperAgent 发起:                                                      │
│   ├── 自然语言解析 → 能力链编排                                         │
│   ├── 跨场景协调 → 能力组合执行                                         │
│   └── 涌现能力 → 动态创建能力                                           │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 1.3 当前实现的用户逻辑

| 用户动作 | 当前实现 | 发起者 | 实现位置 |
|----------|----------|--------|----------|
| 选择模板部署 | ✅ 已实现 | User | SceneTemplateController |
| 创建场景 | ✅ 已实现 | User | SceneService.create() |
| 激活场景 | ✅ 已实现 | User | SceneService.activate() |
| 绑定能力 | ✅ 已实现 | User | CapabilityBindingService.bind() |
| 调用能力 | ⚠️ 部分 | User/Agent | 缺少CapabilityInvoker |
| 工作流执行 | ⚠️ 部分 | Agent | SceneWorkflowService |

| 涌现能力 | ❌ 未实现 | SuperAgent | - |

## 二、发现闭环分析

### 2.1 规格要求的发现闭环

```
用户选择模板
      ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  Phase 1: 安装检查                                                       │
│  ├── 检查 Skills 是否已安装                                              │
│  ├── 未安装 → 依赖解析 → 拓扑排序 → 安装                                  │
│  └── 已安装 → 跳过                                                       │
├─────────────────────────────────────────────────────────────────────────┤
│  Phase 2: 场景创建                                                       │
│  ├── 创建 SceneDefinition                                                │
│  └── 绑定能力需求                                                        │
├─────────────────────────────────────────────────────────────────────────┤
│  Phase 3: 场景组创建                                                     │
│  ├── 创建 SceneGroupInfo                                                 │
│  ├── 添加参与者                                                          │
│  └── 绑定能力实例                                                        │
├─────────────────────────────────────────────────────────────────────────┤
│  Phase 4: 场景激活                                                       │
│  ├── 启动服务                                                            │
│  ├── 建立通信                                                            │
│  └── 状态同步                                                            │
└─────────────────────────────────────────────────────────────────────────┘
      ↓
返回部署结果
```

### 2.2 当前实现对比

| 阶段 | 规格要求 | 当前实现 | 差距 |
|------|----------|----------|------|
| **Phase 1: 安装检查** | 检查→解析→安装 | ✅ SceneTemplateService | 完整 |
| **Phase 2: 场景创建** | 创建SceneDefinition | ✅ SceneService.create() | 完整 |
| **Phase 3: 场景组创建** | 创建SceneGroupInfo | ⚠️ 需手动调用 | 未自动创建 |
| **Phase 4: 场景激活** | 启动服务+通信 | ⚠️ 仅状态变更 | 缺少服务启动 |

### 2.3 发现闭环差距

| 差距项 | 影响 | 优先级 |
|--------|------|--------|
| 场景组未自动创建 | 用户需手动创建场景组 | P0 |
| 服务启动未实现 | 场景激活后服务未真正启动 | P1 |
| 通信链路未建立 | Agent间无法通信 | P1 |

## 三、执行闭环分析

### 3.1 规格要求的执行闭环

```
能力调用请求
      ↓
┌─────────────────────────────────────────────────────────────────────────┐
│  1. 解析capId                                                            │
│     └── 查找 CapabilityBinding，获取 capabilityId                        │
├─────────────────────────────────────────────────────────────────────────┤
│  2. 获取执行组件                                                         │
│     ├── agentId: 执行Agent                                               │
│     ├── linkId: 通信链路                                                 │
│     └── capAddress: CAP地址                                              │
├─────────────────────────────────────────────────────────────────────────┤
│  3. 权限检查                                                             │
│     └── capAddress.isAccessibleFrom(sourceDomain)                        │
├─────────────────────────────────────────────────────────────────────────┤
│  4. 链路检查                                                             │
│     └── link.status == ACTIVE                                            │
├─────────────────────────────────────────────────────────────────────────┤
│  5. 执行调用                                                             │
│     └── agent.invokeByAddress(capAddress, params)                        │
├─────────────────────────────────────────────────────────────────────────┤
│  6. 故障处理                                                             │
│     ├── 链路故障 → 降级绑定                                              │
│     ├── Agent故障 → 故障转移                                             │
│     └── 执行超时 → 重试或返回默认值                                       │
├─────────────────────────────────────────────────────────────────────────┤
│  7. 更新统计                                                             │
│     ├── link.addBytesSent()                                              │
│     └── link.addBytesReceived()                                          │
└─────────────────────────────────────────────────────────────────────────┘
      ↓
返回执行结果
```

### 3.2 当前实现对比

| 步骤 | 规格要求 | 当前实现 | 差距 |
|------|----------|----------|------|
| 1. 解析capId | CapabilityBinding查找 | ✅ CapabilityBindingService | 完整 |
| 2. 获取执行组件 | Agent + Link + Address | ⚠️ 仅存储ID | 缺少运行时获取 |
| 3. 权限检查 | CAP地址权限检查 | ❌ 未实现 | 缺少权限验证 |
| 4. 链路检查 | Link状态检查 | ❌ 未实现 | 缺少链路管理 |
| 5. 执行调用 | agent.invokeByAddress | ❌ 未实现 | 缺少CapabilityInvoker |
| 6. 故障处理 | 降级+转移+重试 | ✅ FallbackService | 已实现 |
| 7. 更新统计 | 流量统计 | ⚠️ 部分 | 缺少实时统计 |

### 3.3 执行闭环差距

| 差距项 | 影响 | 优先级 |
|--------|------|--------|
| CapabilityInvoker未实现 | 无法真正执行能力调用 | P0 |
| Agent运行时管理缺失 | 无法获取执行Agent | P0 |
| Link链路管理缺失 | 无法检查通信状态 | P1 |
| CAP权限检查缺失 | 安全风险 | P1 |

## 七、改进建议

### 7.1 P0 优先级改进

#### 7.1.1 实现CapabilityInvoker

```java
public interface CapabilityInvoker {
    Object invoke(String capId, Map<String, Object> params);
    Object invokeWithFallback(String capId, Map<String, Object> params);
    CompletableFuture<Object> invokeAsync(String capId, Map<String, Object> params);
}
```

#### 7.1.2 实现场景组自动创建

```java
// SceneTemplateService.deployTemplate() 中增加
if (template.getSpec().getSceneConfig() != null) {
    SceneGroupConfigDTO groupConfig = new SceneGroupConfigDTO();
    groupConfig.setName(result.getSceneId() + "-group");
    SceneGroupDTO group = sceneGroupService.create(templateId, groupConfig);
    result.setSceneGroupId(group.getSceneGroupId());
}
```

public interface AgentManager {

    void destroyAgent(String agentId);

public interface LinkStore {
