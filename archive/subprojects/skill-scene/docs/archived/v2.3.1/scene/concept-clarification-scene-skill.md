# 概念梳理：场景技能分类体系

## 一、核心设计理念

### 1.1 两大基石

| 理念 | 含义 | 影响 |
|------|------|------|
| **场景是一种特殊的技能** | 场景技能是技能的子集，具有额外属性 | 分类体系以技能为中心 |
| **技能驱动为中心** | 所有分类、类型都服务于技能 | 场景技能类型是技能属性，不是独立分类 |

---

## 二、概念定义

### 2.1 技能 (Skill)

**定义**: 提供特定能力的可安装、可运行的软件单元。

**分类维度**:

| 维度 | 名称 | 说明 | 示例 |
|------|------|------|------|
| **分类 (category)** | 技能分类 | 技能库的组织分类 | `org`, `vfs`, `llm`, `knowledge`, `sys` |
| **类型 (type)** | 技能类型 | 技能的运行形态 | `service-skill`, `provider-skill`, `scene-skill` |
| **所有权 (ownership)** | 所有权 | 技能的管理归属 | `platform`, `enterprise`, `personal` |

### 2.2 场景技能 (Scene Skill)

**定义**: 具有业务语义、可驱动业务流程的特殊技能。

**判断标准** (代码中的四项标准):

| 标准 | 名称 | 条件 |
|------|------|------|
| 标准1 | 场景技能标识 | `sceneSkill=true` 或 `type=scene-skill` |
| 标准2 | 能力声明 | `sceneCapabilities` 非空 |
| 标准3 | 自驱能力 | `mainFirst && mainFirstConfig && driverConditions` |
| 标准4 | 业务语义 | `driverConditions` 或 `participants` 非空 |

### 2.3 场景技能类型 (Skill Type)

**定义**: 场景技能的激活模式分类，是技能的**属性**而非独立分类。

| 类型 | 名称 | 判定条件 | 特征 |
|------|------|----------|------|
| **ABS** | 自驱业务场景 | 自驱=true, 评分≥8 | 自动驱动，高业务语义，公开可见 |
| **ASS** | 自驱系统场景 | 自驱=true, 评分<8 | 自动驱动，低业务语义，内部可见 |
| **TBS** | 触发业务场景 | 自驱=false, 评分≥8 | 需要触发，高业务语义，公开可见 |

## 四、对比分析

### 4.1 现有设计 vs 正确设计

| 维度 | 现有设计 (错误) | 正确设计 |
|------|----------------|----------|
| **category 含义** | 场景技能类型 (abs/tbs/ass) | 技能分类 (knowledge/llm/org) |
| **skillType** | 不存在 | 场景技能类型 (abs/tbs/ass) |
| **分类关系** | 场景和技能平行 | 场景是技能的子集 |
| **分类来源** | 手动指定 | 自动计算 (基于属性) |

### 4.2 数据结构对比

**现有设计 (错误)**:
```yaml
- skillId: skill-document-assistant
  category: abs              # ❌ 场景类型被误作分类
  subCategory: knowledge     # ❌ 实际分类被降级为子分类
  tags: [...]
  mainFirst: true
```

**正确设计**:
```yaml
- skillId: skill-document-assistant
  category: knowledge        # ✅ 技能分类
  sceneSkill: true           # ✅ 是场景技能
  skillType: abs             # ✅ 场景技能类型 (可自动计算)
  
  # 场景技能属性
  mainFirst: true
  mainFirstConfig: {...}
  driverConditions: [...]
  participants: [...]
  visibility: public
```

### 4.3 分类计算逻辑对比

**现有设计**: 无计算逻辑，手动指定 `category: abs`

**正确设计**: 基于属性自动计算
```
1. 判断是否场景技能: sceneSkill=true 或 type=scene-skill
2. 判断自驱能力: mainFirst && mainFirstConfig && driverConditions
3. 计算业务语义评分: driverConditions(3) + participants(3) + visibility(2) + ...
4. 确定类型:
   - 自驱 + 高评分 → ABS
   - 自驱 + 低评分 → ASS
   - 非自驱 + 高评分 → TBS
```

## 五、讨论点

### 5.1 讨论点一：skillType 应该手动指定还是自动计算？

**方案A: 手动指定**
- 优点: 简单直接，开发者可控
- 缺点: 可能与实际属性不符，需要验证

**方案B: 自动计算**
- 优点: 与代码逻辑一致，保证准确性
- 缺点: 需要完整属性数据

**建议**: 自动计算 + 手动覆盖
```yaml
skillType: abs              # 可手动指定
skillTypeAuto: true         # 或启用自动计算
```

### 5.2 讨论点二：普通技能和场景技能如何区分？

**方案A: 通过 type 字段**
```yaml
type: service-skill         # 普通技能
type: scene-skill           # 场景技能
```

**方案B: 通过 sceneSkill 标志**
```yaml
sceneSkill: false           # 普通技能
sceneSkill: true            # 场景技能
```

**建议**: 两者结合，兼容处理
```yaml
type: scene-skill           # 类型声明
sceneSkill: true            # 显式标志 (可选)
```

### 5.3 讨论点三：分类体系是否需要调整？

**现有分类**:
```
org, vfs, llm, knowledge, sys, msg, ui, payment, media, util, nexus-ui
```

**问题**:
- `nexus-ui` 是否应该独立？
- `llm` 和 `knowledge` 是否应该合并？

**建议**: 保持现有分类，通过 `skillType` 区分场景技能

### 5.4 讨论点四：场景技能类型 (ABS/ASS/TBS) 的实际意义？

| 类型 | 使用场景 | 示例 |
|------|----------|------|
| **ABS** | 自动启动的业务流程 | 文档问答助手、新人培训助手 |
| **ASS** | 后台自动运行的系统任务 | 日志清理、数据同步 |
| **TBS** | 需要人工触发的业务流程 | 会议纪要整理、项目知识沉淀 |

**问题**: 当前 `skill-index.yaml` 中的场景技能是否符合这些定义？

| 概念 | 定义 | 层级 | 示例 |
