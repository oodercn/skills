# 技能分类框架讨论

## 一、分类目的分析

### 1.1 当前问题

| 问题 | 说明 |
|------|------|
| 目的不清晰 | 分类是为了什么？安装？管理？发现？ |
| 维度混杂 | 功能分类、业务领域、能力类型混在一起 |
| 层次不清 | 系统能力、普通能力、扩展能力没有区分 |

### 1.2 分类应该回答的问题

1. **谁使用？** → 系统组件 vs 最终用户
2. **怎么用？** → 独立安装 vs 场景激活 vs 内部调用
3. **做什么？** → 功能类型 vs 业务领域

## 三、整合方案

### 方案一：三层分类体系

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        三层分类体系                                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   第一层：能力层级 (tier) - 回答"谁使用？"                                       │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │  SYSTEM      - 系统能力：安装器、场景管理器、核心服务                      │   │
│   │  STANDARD    - 标准能力：通用功能服务                                    │   │
│   │  EXTENDED    - 扩展能力：第三方集成、业务定制                              │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   第二层：能力类型 (type) - 回答"怎么用？"                                       │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │  STANDALONE  - 独立能力：可独立安装和运行                                 │   │
│   │  INTERNAL    - 内部能力：被其他能力调用，不独立运行                        │   │
│   │  SCENE       - 场景技能：需要场景激活，面向最终用户                         │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   第三层：功能分类 (category) - 回答"做什么？"                                   │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │  KNOWLEDGE   - 知识管理                                                 │   │
│   │  LLM         - AI模型                                                   │   │
│   │  DATA        - 数据存储                                                 │   │
│   │  COMMUNICATION - 通讯服务                                               │   │
│   │  WORKFLOW    - 业务流程                                                 │   │
│   │  SECURITY    - 安全服务                                                 │   │
│   │  MONITOR     - 监控运维                                                 │   │
│   │  TOOL        - 工具服务                                                 │   │
│   │  UI          - 界面服务                                                 │   │
│   │  ORG         - 组织管理                                                 │   │
│   │  IOT         - 物联网                                                   │   │
│   │  COLLABORATION - 协作服务                                               │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   补充维度：业务领域 (domain) - 可选                                            │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │  core, ai, storage, messaging, payment, media, hr, iot, nexus        │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

│   │  WORKFLOW    - 业务流程    │  ORG         - 组织管理                   │   │
│   │  IOT         - 物联网      │  COLLABORATION - 协作服务                 │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   标签：业务领域 (domain) - 可选                                               │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │  core, ai, storage, messaging, payment, media, hr, iot, nexus        │   │
│   └─────────────────────────────────────────────────────────────────────┘   │

## 五、推荐方案：方案三 (简化双维度)

### 5.1 核心概念

```
能力类型 (type):
├── PROVIDER  - 提供者：独立运行，提供能力
├── SCENE     - 场景：面向用户，需要激活
└── INTERNAL  - 内部：被调用，不独立运行

功能分类 (category):
├── KNOWLEDGE     - 知识管理
├── LLM           - AI模型
├── DATA          - 数据存储
├── COMMUNICATION - 通讯服务
├── WORKFLOW      - 业务流程
├── SECURITY      - 安全服务
├── MONITOR       - 监控运维
├── TOOL          - 工具服务
├── UI            - 界面服务
├── ORG           - 组织管理
├── IOT           - 物联网
└── COLLABORATION - 协作服务
```

### 5.2 技能分类示例

```yaml
# PROVIDER 示例
- skillId: skill-llm-openai
  name: OpenAI LLM Provider
  type: PROVIDER
  category: LLM
  domain: ai
  
# SCENE 示例
- skillId: skill-document-assistant
  name: 智能文档助手
  type: SCENE
  category: KNOWLEDGE
  domain: ai
  sceneType: AUTO
  visibility: PUBLIC
  
# INTERNAL 示例
- skillId: skill-llm-context-builder
  name: 上下文构建服务
  type: INTERNAL
  category: LLM
  domain: ai
```

### 5.3 分类统计

| type | 数量 | 说明 |
|------|:----:|------|
| **PROVIDER** | ~80 | 独立运行的服务 |
| **SCENE** | ~10 | 面向用户的场景 |
| **INTERNAL** | ~40 | 内部调用的能力 |

## 六、讨论问题

### 6.1 核心问题

1. **"系统能力"如何定义？**
   - 是否需要区分系统核心能力和普通能力？
   - 安装器、场景管理器是否应该有特殊标记？

2. **"扩展能力"如何定义？**
   - 第三方集成算扩展吗？
   - 业务定制算扩展吗？

3. **能力类型是否足够？**
   - PROVIDER / SCENE / INTERNAL 是否覆盖所有情况？
   - 是否需要增加 PLUGIN / EXTENSION 类型？

### 6.2 建议讨论

| 问题 | 选项A | 选项B | 选项C |
|------|-------|-------|-------|
| 系统能力区分 | 需要区分 | 不需要区分 | 用标签标记 |
| 扩展能力定义 | 第三方集成 | 业务定制 | 两者都是 |
| 能力类型 | PROVIDER/SCENE/INTERNAL | 增加 PLUGIN | 增加 EXTENSION |
